# Project Plan: QR-Based Loyalty Point Application

## 1. Project Overview
A web application allowing customers to scan a QR code, input their order details, and claim loyalty points. The system includes an admin portal to configure dynamic point-calculation logic and track user transactions. Initially, the system will use mocked order data until a third-party API integration is finalized.

## 2. Tech Stack & Tooling
*   **Frontend & Backend:** Next.js (App Router) leveraging React 18/19 hooks (e.g., `useTransition`, `useActionState` for Server Actions).
*   **Database & Authentication:** Supabase (PostgreSQL, Supabase Auth, RLS).
*   **UI/Styling:** Tailwind CSS + shadcn/ui (targeting a minimalist, clean, and responsive design).
*   **Form Validation:** React Hook Form + Zod.
*   **API Testing & Dev Tools:** Cursor IDE for optimized full-stack development; Bruno for testing and mocking API endpoints.

---

## 3. Database Architecture
Core tables in Supabase:
1.  **`point_settings`**: Stores dynamic logic (e.g., conversion rate, minimum order value).
2.  **`orders`**: Stores order information. Currently populated with mock data; eventually synced via third-party webhooks.
3.  **`customers`**: Stores user identity (`phone`, `email`, `full_name`) and `total_points`.
4.  **`point_transactions`**: Ledger for all points claimed.

---

## 4. Implementation Phases

### Phase 1: Environment Setup & Initialization
*   Initialize Next.js project with TypeScript and Tailwind CSS.
*   Set up Supabase project and establish a database connection via `@supabase/ssr`.
*   Execute SQL scripts to create schemas and apply Row Level Security (RLS) policies.
*   Seed `orders` table with mock data to unblock frontend development.

### Phase 2: Customer Portal (QR Web App)
*   **UI Construction:** Build a clean, minimalist form interface using shadcn/ui components.
*   **Step 1 - Order Validation:** 
    *   Create an input field for the `order_code`.
    *   Implement a Next.js Server Action to query the `orders` table and calculate potential points on-the-fly based on the active `point_settings`.
*   **Step 2 - User Information:** 
    *   Gather `full_name`, `email`, and `phone`. 
    *   Implement debounce logic to fetch existing customer data via phone number.
*   **Step 3 - Transaction Execution:** 
    *   Submit the complete payload.
    *   Perform a database transaction: update `orders` status to 'claimed', increment customer's `total_points`, and insert a record into `point_transactions`.

### Phase 3: Admin Portal
*   **Authentication:** Implement login page using Supabase Auth (Email/Password).
*   **Dashboard Layout:** Build a protected route layout (`/admin`).
*   **Point Configuration Module:** 
    *   Create a setup page to manage `point_settings` (e.g., updating conversion rates).
*   **Data Management Tables:**
    *   Implement interactive data tables for "Customer List" and "Transaction History".

### Phase 4: Third-Party Integration & Testing
*   **API Mocking:** Utilize Bruno to simulate incoming webhooks from the third-party order system.
*   **Adapter Implementation:** Create a dedicated API route (`/api/webhooks/orders`) to receive, validate (via Zod), and standardize external order data before inserting it into the `orders` table.

### Phase 5: CI/CD & Deployment
*   **Testing Environment:** Deploy the initial release to the testing environment (`mia`) for QA and flow verification.
*   **Production Deployment:** Promote the stable build to the production environment (`EVA`).
*   Configure environment variables securely across all deployment stages.
